I Quit Tmux

Oct 11, 2024

In June 2023 I installed Wezterm and a month later I quit Tmux.
It’s been more than a year since I quit, and I can confidently say that I won’t go back.
In this blog post I will go through why I replaced my workflow in Tmux with Wezterm, and the pros and cons of doing so.

How did I use Tmux?

Panes & Windows

The main usecase I had from Tmux was to write my code in Neovim, and then run the code in a split pane.
In doing so I would usually have my Neovim instance running in a zoomed pane, and then unzoom to compile or whatever I would need a terminal for.
If I were to use something like Lazygit or quickly edit a file in another directory, then I would use either a floating pane or a new Tmux window (tab for regular folks) as to not interrupt/pollute my neovim, split layout.

Sessions

As a CS student I would also frequently switch context throughout the day/week, as I would be writing a compiler in one course and a matrix multiplication algorithm in another. As such I would switch git repositories frequently. For doing this I used my favorite Tmux plugin t, a predecessor of sesh. This plugin would use fzf and zoxide to quickly create/connect to Tmux sessions with the session name and cwd set to a chosen directory. This made it super quick to switch context, and changed the way I work in the terminal.

Why did I switch?

But why switch from Tmux? I though Tmux is what all the cool h4ck3rs are using?
Tmux is amazing, and I don’t have many reasons for making the switch, but here are some:

Not having to start/attach to Tmux

One issue I was having when I used Tmux was the problem of opening my terminal, navigating to my repository, starting Neovim, getting in the zone, quickly splitting my pane to compi… Oh, wait… I forgot to attach to Tmux, meaning I now have to open everything again inside a Tmux session, pulling me out of the zone…
To fix this issue, one can simply make starting and attaching to a Tmux session a part of spawning a shell. Sure, it works, but is a tiny bit slower whenever you open a shell and I wasn’t a big fan of this.

Wezterm however does not suffer from this problem. In Wezterm you can simply split panes, spawn tabs ect. as it is attached to a default Session (Workspace) whenever you start the terminal. This seems like a minor inconvinience, but was actually the main reason that I made the switch to Wezterm.

Tmux configuration sucks

As a programmer I find the Tmux configuration experience a lot worse than Wezterm. Wezterm is configured in Lua which is a very simple, but powerful language and perfect for configurations in my opinion. At least it is easier for me to read and write an actual programming language, than the DSL that is the Tmux configuration + shell scripts.

Below is an example of a Tmux and Wezterm keybind to split the current pane vertically and spawn top in the left pane.

bind-key -n C-t run-shell "tmux split-window -h -p 50 'top' && tmux swap-pane -D"

config.keys = {
  {
    key = 't',
    mods = 'CTRL',
    action = wezterm.action.SplitPane {
      direction = 'Left',
      command = { args = { 'top' } },
      size = { Percent = 50 },
    },
  },
}

I personally find it very hard to understand the Tmux keybind, but that might just be a skill issue…
Furthermore, if you are using Tmux, then you are most likely also using a terminal emulator with a configuration. Combining the multiplexer and terminal configuration into one also improves the simplicity of my dotfiles.

Trying new software is fun

The number one reason that I switched might just be because I was looking into terminal emulators and in my search of finding the best, I found Wezterm. Reading the multiplexing and the Workspaces/Sessions sections of the docs, I figured I could replace my Tmux setup with the features built into Wezterm. So over the summer break I replicated most of the features from Tmux in my Wezterm config, and it was very enjoyable to configure it to my liking.

What are the pros?

Wezterm has a lot of features which I wasn’t aware of before I switched which I only discovered after switching.

The killer feature of Wezterms domains

Wezterm has the concept of domains. Domains are the “context” for which a pane exists. By default you use “local” domains, which are just regular shell sessions. However in Wezterm you can also spawn panes in other domains, like a ssh domain, which will make a pane with an active ssh connection. This pane can then be split, opened in a new tab, ect. Just like a regular “local” pane, but without you having to manually ssh into the server every time. This means that you essentially get the layout features you are used to from Tmux, without having to have Tmux installed on the server. This also works for other domains, like WSL, Unix sockets, TLS and you can even create your own domains, like docker domains.

I am not aware of a feature like this in Tmux, although it is probably possible with some plugin or configuration, but the feature is very well integrated into Wezterm. With some nice keybindings, like quick_domains then it is amazing to work with, especially when you don’t have access to Tmux on a server.

Below is gif of me spawning a pane in a domain which automatically creates a ssh connection to a Raspberry Pi. I am able to freely split panes like I would with regular (“local”) panes and splitting a pane, will spawn a split pane within that specific domain. This is why the left panes both are on my laptop, while the right panes are on the Raspberry Pi.

Great documentation and APIs

I have spent many hours reading through the documentation of Wezterm and all the features are really well documented and most have easy to follow examples. Through reading the docs I have discovered many interesting ways to use features I would otherwise ignore to improve my workflow, like this example or the docker domain from before.

The best maintainer and community in Open Source

Sure, that is a bit of a bold statement, but Wez is a really great maintainer. Not only for his work on Wezterm, but for the way he approaches contributers. He is always very friendly, appreciative and gives detailed feedback on PRs. I aspire to be like him whenever I’m giving feedback.
The community is also really supportive and helpful, whether it is people on Reddit, Matrix or github. I have never seen any communication that left a bad taste in my mouth.

What are the cons?

Not as popular

Wezterm is a newer terminal with a smaller group of users. As such there is less information about Wezterm compared to Tmux.
This also means that some of the plugins from Tmux, like tmux-resurrect or the t plugin I mentioned above, was not available when I switched. This made me develop replacement plugins, like: resurrect and smart_workspace_switcher, however now I don’t find that I am missing any plugins.
And in the last few months a lot of other plugins have been developed, see the awesome-wezterm list.

Missing floating panes

One feature I’m missing from Tmux is the ability to have floating panes. This is not yet a part of Wezterm, but in recent months big progress has been made to implement them (might even be implemented when you read this).
Instead of using floating panes for opening Lazygit or similar, I use tabs as a replacement. While it is almost as good as floating panes, tabs can’t quite replace the seamless experience of quickly toggleing a floating pane. As such I sometimes end up having random unused tabs open cluttering my tab bar.

One multiplexer window per GUI window

Wezterm restricts multiplexer windows to only one GUI window. This means that you unfortunetly cannot have 2 instances of the same multiplexer window attached.
In Tmux you can have 2 sessions opened at the same time, by attaching one session in one terminal window, and another session in another terminal window.
This is unfortunetly not possible in Wezterm, and can sometimes be annoying if you want to change something in one workspace, based on something from another workspace.

Some general differences

Workspaces vs. Sessions

In Wezterm the equivalent to Tmux sessions are workspaces. Workspaces work a lot like sessions, in that they contain the state of the multiplexer for that session. You can switch workspaces just like you can switch sessions, but in Wezterm workspaces also contain windows, aswell as tabs and panes.

Should you quit Tmux?

I am very happy with my desicion to switch to Wezterm. Although there were minor downsides to the switch, I found the features exclusive to Wezterm and the ability to easily update my configuration via. great documentation and Lua a huge win.
So I know that I won’t be going back in the current state, and probably will be even less inclined to do so with the addition of new plugins and features into Wezterm.

But should you make the switch? Honestly both Wezterm and Tmux have really great multiplexing capabilities, and share most features. So I would only advice you to change if you want to spice up your multiplexing life and try new experiences. But if you are on the fence, then you can always use both, by running Tmux inside Wezterm.
